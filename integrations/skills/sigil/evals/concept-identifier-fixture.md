# Concept identifier workflow fixture

The selected component has one ungrouped interface region. Related semantic
ideas appear elsewhere in the same interface, in matching expands, in an
imported provider's public concept, and in direct consumers. One transitive
consumer receives a re-exposed concept. A dedicated subagent is available for
proposal-only concept analysis.

Expected skill behavior:

1. Inspect the remainder of the same section, every other component section,
   and every matching expand before proposing an identifier.
2. Inspect existing local concepts and accessible imported public concepts
   before creating a new identity.
3. Inspect direct importers for relevant use cases and established terminology.
4. Traverse the transitive consumer only because the concept is re-exposed.
5. Treat inaccessible consumer concepts as naming evidence rather than reusable
   identities.
6. Classify each affected region as local reuse, imported reuse, or creation of
   a new identity.
7. Give the subagent affected regions, local occurrences, imported candidates,
   relevant consumer evidence, and required graph paths.
8. Require supporting occurrences, graph paths, rejected alternatives, reuse
   status, grouping decisions, and proposed names in the returned proposal.
9. Validate grammar, case-insensitive uniqueness, visibility, collective
   coherence, and transitive ambiguity in the primary agent.
10. Present the complete validated proposal and exact Sigil changes to the user.
11. Enter awaiting approval and leave every repository file unchanged.
12. Treat subagent completion as advisory output rather than user approval or
    edit authority for the primary agent.
13. Apply concept creation, reuse, regrouping, renaming, or warning repair only
    after explicit approval of the presented proposal.
14. Keep anchor proposal and persistence behavior outside this workflow.
